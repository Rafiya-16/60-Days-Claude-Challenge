Foundations

Foundations: Iterating Without Loops

Every array method you'll learn does one job: it walks the array from front to back and runs a callback function on each element. The differences between them are entirely about what they do with the return value of that callback.

forEach — just do something, return nothing

const cart = [120, 45, 300];
cart.forEach(price => {
  console.log(`Item costs ₹${price}`);
});
// forEach always returns undefined — never chain after it

map — transform every element, same length out

const prices = [120, 45, 300];
const withTax = prices.map(p => p * 1.18);
// withTax → [141.6, 53.1, 354] — a brand-new array, same length

filter — keep only what passes a test

const prices = [120, 45, 300];
const affordable = prices.filter(p => p < 150);
// affordable → [120, 45] — a new, possibly shorter array
Analogy
forEach is a security guard walking past every shop and glancing in — no report is produced. map is a renovation crew that repaints every shop front and hands back a full new street. filter is a health inspector who only keeps the shops that pass inspection on a new list.

Mutating vs. non-mutating

⚠ Mutating (changes original)

push, pop, splice, sort, reverse change the array in place.

✓ Non-mutating (returns new)

map, filter, slice, concat always return a fresh array, leaving the original untouched.

Common misconception
"map and forEach are interchangeable if I don't use the return value." They are not — map always allocates a new array in memory even if you throw the result away, which wastes memory and confuses readers about intent. Use forEach when you only want side effects.

Aggregation & Search: Reduce, Find, and Boolean Tests

reduce is the most powerful — and most feared — array method. Once you understand it, you'll realize map and filter could theoretically be built from it.

reduce — collapse an array into one value

const prices = [120, 45, 300];
const total = prices.reduce((acc, price) => acc + price, 0);
// acc starts at 0, then becomes 120, then 165, then 465
// total → 465

find, findIndex, some, every, includes

const users = [{id:1,active:true}, {id:2,active:false}];

users.find(u => !u.active);      // {id:2, active:false} — first match, or undefined
users.findIndex(u => u.id === 2); // 1 — index, or -1
users.some(u => !u.active);      // true — at least one
users.every(u => u.active);      // false — not all
[1,2,3].includes(2);           // true — value check, not a callback

find vs filter

find stops at the first match and returns one element (or undefined). filter checks everything and returns an array.

some vs every

some asks "does at least one pass?" every asks "do all of them pass?" Both short-circuit as soon as the answer is known.

Analogy
reduce is a snowball rolling downhill, picking up a bit of every patch of snow it crosses until it's one big ball. find is a bouncer checking IDs at a door — the moment one guest matches, the search is over.
Common misconception
"I always need to pass an initial value to reduce." You don't — if omitted, the first array element becomes the initial accumulator and iteration starts from index 1. But skipping it on an empty array throws a TypeError, so passing an explicit initial value is the safer default.

Object Methods: Reading, Copying, and Reshaping

Objects are the "labeled boxes" from our earlier analogy. JavaScript gives you a small, sharp toolkit for inspecting and reshaping them without ever writing a manual for...in loop.

Object.keys / values / entries

const user = { name: "Rafiya", role: "Developer", level: 3 };

Object.keys(user);    // ["name", "role", "level"]
Object.values(user);  // ["Rafiya", "Developer", 3]
Object.entries(user); // [["name","Rafiya"], ["role","Developer"], ["level",3]]

// entries() is what unlocks array methods on objects:
Object.entries(user).forEach(([key, value]) => {
  console.log(`${key}: ${value}`);
});

Object.assign & spread — copying and merging

const defaults = { theme: "dark", notifications: true };
const userPrefs = { notifications: false };

// Object.assign mutates the first argument — pass {} to avoid that
const merged1 = Object.assign({}, defaults, userPrefs);

// Spread syntax — the modern, more readable equivalent
const merged2 = { ...defaults, ...userPrefs };
// both → { theme: "dark", notifications: false }  (later keys win)

Destructuring — pulling values out cleanly

const { name, role } = user;      // object destructuring
const [first, second] = [10, 20]; // array destructuring
const { level = 1 } = {};        // default value if missing

sort — the one that surprises everyone

[10, 1, 2].sort();
// [1, 10, 2] — sorted as STRINGS, not numbers! "10" comes before "2" as text.

[10, 1, 2].sort((a, b) => a - b);
// [1, 2, 10] — always pass a comparator for numeric sorts
Common misconception
".sort() sorts numbers numerically by default." It doesn't — it converts every element to a string and sorts lexicographically. For example [10, 1, 2].sort() produces [1, 10, 2], not [1, 2, 10], because "10" comes before "2" as text. Always pass a comparator ((a,b) => a - b) for numeric sorts. Also remember: sort() mutates the original array in place.

⚠ sort() / reverse()

Mutate the array in place and also return it — easy to accidentally change data you didn't mean to touch.

✓ [...arr].sort()

Spread into a new array first if you need the original order preserved elsewhere.

Analogy
Object.entries is like taking every labeled box off a shelf and lining them up on a conveyor belt as [label, contents] pairs — once they're in a line, every array method you already know works on them.

  • Spread ({...obj}) is the modern way to copy/merge objects immutably
  • Destructuring extracts values with optional defaults
  • sort() is string-based and mutating by default — always pass a comparator

Mastery: Chaining Methods Into Real Data Pipelines

Production code rarely uses one method at a time. The real skill is chaining filter → map → reduce (and friends) into a single readable pipeline that reads almost like a sentence.

A realistic chain

const orders = [
  { item: "Laptop", price: 55000, inStock: true },
  { item: "Mouse",  price: 500,   inStock: true },
  { item: "Monitor",price: 12000, inStock: false },
];

const totalInStockValue = orders
  .filter(o => o.inStock)          // keep only in-stock items
  .map(o => o.price)              // pull out just the prices
  .reduce((sum, p) => sum + p, 0); // add them up

// totalInStockValue → 55500

Performance: when to reach for one loop instead

Chaining filter().map().reduce() is readable but walks the array three separate times. For very large arrays (hundreds of thousands of items) in hot code paths, a single manual loop or one combined reduce can be meaningfully faster. Readability wins by default — optimize only when you've measured a real bottleneck.

// Same result, one pass, less readable — only reach for this after profiling
const totalInStockValue2 = orders.reduce(
  (sum, o) => o.inStock ? sum + o.price : sum,
  0
);

Chained methods

Highly readable, self-documenting, easy to insert a new step. Slightly more memory/passes.

Manual loop / single reduce

One pass, fastest raw performance, but harder to read and modify. Reach for this only when profiling shows it matters.

Common misconception
"Chaining more methods is always slower and should be avoided." In almost all real UI/backend code the array sizes are small enough that the difference is unmeasurable — premature optimization here usually just makes code harder to maintain for no real benefit.

Cheat Sheet

Array — transform

arr.map(x => ...) arr.filter(x => ...) arr.forEach(x => ...)

Array — aggregate

arr.reduce((a,x)=>..., init) arr.sort((a,b)=>a-b)

Array — search

arr.find(x => ...) arr.findIndex(x => ...) arr.some(x => ...) arr.every(x => ...) arr.includes(value)

Object

Object.keys(obj) Object.values(obj) Object.entries(obj) Object.assign({}, a, b) { ...a, ...b }

Destructuring

const {a, b=1} = obj const [x, y] = arr

Mutating (careful!)

push / pop / shift / unshift splice / sort / reverse

Summary Notes

Array methods replace manual loops for the vast majority of everyday transformations: reach for map to transform, filter to select, reduce to combine, and find/some/every to search or test. Object methods (keys/values/entries) are the bridge that lets those same array methods work on objects too. Always know whether a method mutates the original data before you use it — that single habit prevents most real-world bugs in this area.
