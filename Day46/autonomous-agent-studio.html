        Architecture overview — how the stop-check governs information flow
        

          
Each round is a real network round-trip, not a fixed schedule:

          
    
            
  1. Planner runs once, turning your spec into a structured build plan (approach, edge cases to cover, function signature).
    2. 
            
  2. Executor runs once, producing draft v1 from that plan.
    3. 
            
  3. The loop begins: Evaluator scores the current draft against the fixed rubric (correctness, readability, efficiency, edge cases) and returns a numeric score + written rationale — always as literal model output, never computed by JS.
    4. 
            
  4. Critic receives the draft + evaluation and produces a targeted list of concrete flaws / risks, independent of the score itself (adversarial second opinion).
    5. 
            
  5. Improver receives the draft + evaluation + critique and rewrites the draft, addressing the specific points raised. Its output becomes next round's draft.
    6. 
            
  6. Memory Manager appends {round, score, delta, critique summary} to the running history array, which is threaded into every subsequent Evaluator/Improver prompt so agents have full context, not just the latest round.
    7. 
            
  7. After Memory Manager logs the round, the stop-check runs in strict order: plateau (score gain < 0.3 for 2 consecutive rounds) → threshold (score ≥ 9.0) → hard cap (12 rounds, safety fallback only). Whichever fires first ends the loop and is what you see reported as "stop reason."
    8. 
            
  8. Only once a stop condition fires does control branch to Final Reviewer, which reads the entire history and produces closing notes — it never runs mid-loop.
    9. 
          

        

      


      

        Extension ideas & further prompts
        

          
    
            
  • Add a Safety Monitor agent that screens the draft for unsafe patterns (eval(), unsanitized SQL, secrets) before Executor output ever reaches Evaluator.
    • 
            
  • Swap the Improver's rewrite strategy: instead of full rewrite, have it emit a unified diff and apply it programmatically — cheaper tokens, clearer audit trail.
    • 
            
  • Parallelize Evaluator + Critic within a round (they don't depend on each other) to cut latency roughly in half.
    • 
            
  • Persist the history array to localStorage so a long run can resume after a page reload.
    • 
            
  • Try prompting Planner to output multiple candidate approaches, then have Executor build the top 2 and let Evaluator pick a winner before the refinement loop starts.
    • 
            
  • Generalize this shell beyond code: swap the rubric + agent prompts and the same loop/stop-check machinery drives a writing, research, or data-cleaning agent unchanged.
